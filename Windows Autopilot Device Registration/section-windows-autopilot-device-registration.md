# Windows Autopilot Device Registration

## Internal Operational Reference

## Objective

Ensure Windows devices are registered with Windows Autopilot so that provisioning through Windows Out-of-Box Experience (OOBE) is automated and tenant-controlled.

## High-Level Flow

Autopilot registration follows a consistent sequence. Device hardware identities are obtained, those identities are registered in Microsoft Intune, provisioning intent is associated using an Autopilot deployment profile, and provisioning is triggered only when the device enters Windows OOBE.

## Autopilot Registration

Registering a device with Windows Autopilot associates the physical device with the organization’s Microsoft Entra tenant using a hardware hash. This process does not enroll the device, apply configuration, install applications, or modify the operating system. Registration only enables device recognition during OOBE.

## Path 1: OEM-Provided Hardware Hash

The preferred approach is to obtain a hardware hash CSV directly from the OEM. The CSV contains the identifiers required for Autopilot, including serial number, manufacturer, model, and hardware hash. Once imported into Intune under Windows Autopilot devices, the devices become registered with the tenant. After import, devices typically appear with identifying metadata and a deployment profile status of Not assigned. This state indicates the device is registered but inactive.

## Path 2: Manual Hardware Hash Collection

When OEM-provided data is unavailable, hardware hashes may be collected manually using a supported PowerShell script. Microsoft provides the Get-WindowsAutopilotInfo script for this purpose. The script queries device-specific hardware identifiers, generates the Autopilot hardware hash, and outputs a CSV compatible with Intune import. The output includes a signature hash used to validate the integrity of the hardware identity.

## PowerShell Execution Context

PowerShell execution policies may block script execution by default. The execution policy must allow scripts to run in order to collect the hardware hash. This change is local to the device and does not affect tenant-level security controls.

## PowerShell Commands

The current execution policy can be checked using Get-ExecutionPolicy. Script execution may be enabled using Set-ExecutionPolicy Unrestricted. The Autopilot script is installed from the PowerShell Gallery using Install-Script -Name Get-WindowsAutopilotInfo. The hardware hash CSV is generated by running Get-WindowsAutopilotInfo.ps1 with an output file specified.

## Hardware Hash Output

The script produces a CSV file containing one row per device. This file can be imported directly into Intune under Windows Autopilot devices. Once imported, the device follows the same lifecycle as an OEM-registered device.

## Deployment Profile Association

Autopilot deployment profiles define device behavior during Windows setup, including join type and user interaction requirements. Profiles may be assigned directly to devices or through Microsoft Entra ID groups. Profile assignment does not enroll the device or trigger provisioning. Profiles are evaluated only when the device enters OOBE.

## Existing Devices

Existing devices require different handling depending on whether a reset is possible.

Devices that can be reset may be registered in Autopilot, assigned a deployment profile, and then wiped or reset. Upon entering OOBE, Autopilot provisioning begins and the device is treated as a new deployment.

Devices that cannot be reset cannot be taken over by Autopilot in their current state. These devices may be manually enrolled into Intune for ongoing management. Registering them in Autopilot ensures automated provisioning during any future reset.

Devices that are already enrolled in Intune but not registered in Autopilot continue to function normally but require manual setup after reset. Registering these devices in Autopilot enables zero-touch provisioning for future lifecycle events.

## Autopilot Provisioning Conditions

Autopilot provisioning occurs only when the device hardware hash is registered in the tenant, a deployment profile is assigned, and the device enters Windows OOBE with network connectivity.

## BYOD Considerations

This Autopilot registration approach applies exclusively to IT-owned Windows devices that require zero-touch provisioning during Windows Out-of-Box Experience. Bring Your Own Device (BYOD) scenarios do not require Windows Autopilot, as user-owned devices are enrolled into Intune through user-driven enrollment after device setup. BYOD devices are not registered with Autopilot and are managed using separate enrollment, compliance, and configuration policies. This separation ensures corporate provisioning controls do not apply to personal devices.

## Operational Model

Autopilot operates across three layers. Registration establishes tenant ownership, deployment profiles define provisioning behavior, and Windows OOBE triggers execution. All three layers must be present for Autopilot provisioning to occur.

## Summary

Windows Autopilot registration is a preparatory identity step that enables automated provisioning during Windows setup. Importing hardware hashes has no immediate effect on devices. Provisioning occurs only during OOBE after a deployment profile is assigned. Existing devices may be incorporated immediately through reset or registered for future redeployment.
